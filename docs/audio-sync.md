# Audio Sync Troubleshooting

## Why Clips Are Re-Encoded

Clips are generated by re-encoding with `libx264` (video) and `aac` (audio) rather than stream copying. This ensures perfect audio/video synchronization and prevents the common issue where video and audio become desynchronized in the output clips.

Use `FFMPEG_PRESET` to control the speed/quality trade-off — see [configuration.md](configuration.md#ffmpeg-preset-guide).

## Problem: Audio Is Delayed or Starts Early

### Symptoms

- Video starts at the correct moment but audio plays 2–5 seconds later or earlier
- Lip movements don't match speech in the clip
- Clip content doesn't match the transcript description

### Root Causes

1. **Transcript misalignment** — transcript timestamps don't perfectly match the video
   - Auto-generated captions often have 1–3 second delays
   - Manual captions are usually more accurate but can still have timing issues
   - Multiple caption tracks (different video versions) can have systematic offsets

2. **Version differences** — the transcript may have been generated from a slightly different version of the video

## Solution: `TIMESTAMP_OFFSET_SECONDS`

Applies a global offset (in seconds) to all clip timestamps before cutting.

- **Positive value** → shifts all clips later
- **Negative value** → shifts all clips earlier

```bash
# In .env
TIMESTAMP_OFFSET_SECONDS=-3

# Or inline
TIMESTAMP_OFFSET_SECONDS=-3 video-clipper <url> --clip
```

## Finding the Correct Offset

### Step 1: Run a single segment with no offset

```bash
TIMESTAMP_OFFSET_SECONDS=0 video-clipper <url> --download-sections 1
```

### Step 2: Play and inspect

- Open the generated clip
- Check if the moment matches the transcript description
- Note whether the clip starts too early or too late

### Step 3: Adjust

If the clip **starts 3 seconds late** (content arrives after it should):

```bash
TIMESTAMP_OFFSET_SECONDS=-3   # shift earlier
```

If the clip **starts 2 seconds early** (content hasn't arrived yet):

```bash
TIMESTAMP_OFFSET_SECONDS=2    # shift later
```

### Step 4: Verify across multiple clips

```bash
TIMESTAMP_OFFSET_SECONDS=-3 video-clipper <url> --download-sections 3
```

Check that the offset works consistently across different segments. If it varies per segment, the issue is video-specific rather than a global transcript offset.

### Binary Search for the Optimal Offset

If you are unsure of the exact value, try a few candidates and narrow down:

```bash
for offset in 0 -3 -6 -9; do
  TIMESTAMP_OFFSET_SECONDS=$offset video-clipper <url> --download-sections 1
  echo "Tested offset: $offset"
done
```

Then narrow down: if `-3` looks close, try `-2` and `-4`.

## Common Scenarios

| Scenario                 | Likely Offset | Explanation                                         |
| ------------------------ | ------------- | --------------------------------------------------- |
| Auto-generated captions  | `-1` to `-3`  | ASR timing often lags slightly behind actual speech |
| Manual captions          | `0` to `-1`   | Usually more accurate, small sync issues possible   |
| Multiple caption tracks  | `-2` to `-5`  | Different versions may have a systematic offset     |
| Regional/dubbed versions | Varies        | Different regions may have different caption timing |

## Verifying the Fix

After applying `TIMESTAMP_OFFSET_SECONDS`:

1. **Watch the clip** — audio and video should be synchronized
2. **Check multiple clips** — the offset should be consistent across segments
3. **Compare with the original** — the clip should match the described moment in the transcript

If the offset varies between segments, the transcript itself may have non-uniform timing and `TIMESTAMP_OFFSET_SECONDS` alone won't fully resolve it.
